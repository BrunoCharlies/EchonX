Qubic, QPI e possibilidade de moeda EchonX

Análise da linguagem de contratos Qubic, do que o app EchonX usa hoje, e de um caminho hipotético para wallet + token + pagamento de assinatura. Decisões registradas

1. Duas “linguagens” no ecossistema

Camada Tecnologia Tipo de restrição
EchonX (app) TypeScript, Next.js, Supabase, Web Speech Regras de produto: planos, ~220 chars na fala, moderação, APIs
Qubic (chain) C++ restrito + QPI (qpi.h), no core Sem ponteiros, sem arrays C, estado em StateData, fees em QU, governança por computors

O repositório EchonX hoje integra Qubic como canal de conteúdo (espelho do X → Supabase), não como smart contract de pagamentos. Stripe + subscriptions.plan cobrem assinatura off-chain.

2. O que a Qubic exige (resumo do contracts.md)

  • Contrato = struct ContractBase + StateData; memória estática (até ~1 GB, mas hash do estado custa por tick).
  • Única API externa: QPI — sem bibliotecas de terceiros.
  • Procedures alteram estado (state.mut()); functions só leem (state.get()).
  • Ativos: qpi.issueAsset() — nome até 7 caracteres (ex.: ECHONX).
  • Deploy: PR no qubic/core, verificação formal, testes, proposta + voto dos computors, IPO, epoch de construção.
  • Contrato precisa queimar QU para reserva de execução — senão para de rodar.

Documentação oficial: Smart Contract Development (Qubic core)

3. Capacidade de implementação (expectativa realista)

O assistente pode: desenhar procedures/functions, seguir restrições do subset C++, alinhar com qpi.h, esboçar testes e spec.
Não substitui: auditoria final de fundos, aprovação do core team, testnet multi-node, nem o processo de votação dos computors.

“Dominar exatamente” o QPI = iterar com o código-fonte do core + ferramenta de verificação + revisão humana experiente em Qubic.

4. Visão: moeda EchonX + wallet + assinatura

Produto desejado (hipótese)

  1. Usuário compra moeda “EchonX” na plataforma.
  2. Usa para pagar assinatura (planos Free / Starter / Popular / Pro).
  3. Wallet na UI para ver e armazenar saldo.

Três modelos possíveis

Modelo On-chain Off-chain Prós Contras
A — Ledger no app Nenhum Saldo em Postgres Rápido; Stripe simples Centralizado; não é cripto nativa
B — Híbrido Contrato recebe QU + procedure PayPlan App desbloqueia plan após tx Pagamento verificável Contrato no core; UX de seed/wallet
C — Asset ECHONX issueAsset + contrato tesouraria Indexer / espelho de saldo Marca forte “moeda da plataforma” Mais complexo que QU puro; fees de estado

Fluxo natural na Qubic (assinatura)

Transaction → contrato com inputType “pagar plano X” → procedure valida invocator() e valor → estado on-chain → EchonX indexa e atualiza subscriptions no Supabase.

Wallet na plataforma

  • Custodial: backend assina transações (compliance pesado).
  • Não custodial: usuário traz seed / wallet Qubic; app só monta transações (melhor alinhamento, mais fricção).

Falta hoje: vínculo explícito profile EchonX ↔ id Qubic (32 bytes).

5. Ordem sugerida de trabalho

  1. Economia: emissão, preço, reembolso, planos.
  2. Protótipo com QU em testnet antes de asset custom.
  3. Contrato mínimo: depositar, pagar plano, consultar saldo (function).
  4. UI wallet + mapeamento de identidade.
  5. IPO / integração core se fizer sentido estratégico.

8. Modelo de produto (síntese)

Funil econômico: QU (Qubic) → compra de ECHONX → consumo (assinatura / serviços). Qubic não é o produto principal; é rampa de entrada para o token da plataforma.

Implicações técnicas (rascunho)

  • Off-chain (EchonX): ledger de créditos ECHONX por usuário, custódia, preços de planos em ECHONX e em QU, Stripe opcional em paralelo até integração chain madura.
  • On-chain (futuro): contrato ou integração Qubic para depósito de QU e emissão/crédito de ECHONX; procedures de pagamento de plano aceitando QU ou asset ECHONX.
  • Não fazer agora: P2P de ECHONX entre usuários; wallet não custodial.
  • Compliance: wallet custodial exige política de custódia, KYC/AML conforme jurisdição, e segurança de chaves no backend.

Modelo alinhado (vs. tabela da seção 4)

Combinação próxima do modelo A (ledger no app) para ECHONX como crédito de consumo, com ponte B para aceitar QU na compra de ECHONX e no pagamento de assinatura. Asset on-chain ECHONX (modelo C) entra quando o contrato Qubic estiver aprovado no core.

9. Por onde começar: carteira ou criptomoeda?

Recomendação: começar pela carteira (ledger) por perfil no EchonX — não pelo smart contract / asset ECHONX na Qubic.

Por quê nesta ordem?

Ordem O quê Motivo
1º Definição da “moeda” como produto (ECHONX crédito) Regras: não transferível (v1), unidade, decimais, quem emite (tesouraria EchonX), preço em QU.
2º Carteira custodial por perfil Saldo, histórico, compra com QU (manual/admin no início), débito na assinatura. Tudo em Postgres + UI — sem IPO Qubic.
3º Checkout de plano com ECHONX e/ou QU (espelho off-chain) Valida receita e UX antes de chain.
4º Asset / contrato ECHONX na Qubic Meses + governança; espelha o ledger quando estiver pronto.

Se começar pela cripto on-chain primeiro

  • IPO, proposta aos computors, fees de execução, estado do contrato — sem usuário final usando por meses.
  • Carteira custodial no app ainda seria necessária para 99% dos usuários (não operam seed Qubic no dia a dia).
  • Risco de desenhar token on-chain que não bate com crédito de consumo off-chain.

Se começar pela carteira primeiro (ideal)

  • Cada profile ganha conta interna: saldo ECHONX, extrato, bloqueios.
  • “Comprar ECHONX com QU” pode ser fase 1 manual (admin credita após ver depósito) → fase 2 API/indexer Qubic → fase 3 contrato.
  • Assinatura já debita ECHONX do ledger; Stripe pode coexistir.
  • Quando o asset ECHONX existir na chain, a carteira vira ponte (depósito/saque), não recomeço do zero.

Metáfora

Primeiro o banco interno e o cartão de consumo (carteira + crédito ECHONX); depois a nota oficial cunhada na rede (asset Qubic). O usuário só precisa ver saldo e pagar plano — não precisa saber em qual fase técnica você está.

10. Carteira primeiro: ECHONX + stablecoin + QU?

Sim. Se a carteira for um ledger custodial por perfil (conta interna), você pode modelar vários tipos de saldo desde o dia 1 — não só ECHONX.

Como pensar a carteira

Não é “uma carteira blockchain” no v1; é uma conta EchonX com linhas de saldo por ativo:

Ativo na UI Papel no produto Custódia real (v1 típico)
ECHONX Crédito de consumo (assinatura, serviços) Ledger Postgres + tesouraria EchonX
QU (Qubic) Entrada para comprar ECHONX; pagar plano (decisão: ambos) Carteira custodial EchonX com endereço/ID Qubic da plataforma ou sub-conta por usuário (fase 2+)
Stablecoin (ex. USDC) Opcional: depósito em dólar, preço estável, menos volatilidade Depósito on-chain em endereço custodial ou parceiro (CEX/API) — não é automático só por criar ledger

Respostas diretas

  • Saldo em stablecoin? Sim, se você adicionar o tipo de ativo no schema e definir como entra/sai (rede, parceiro, conversão para ECHONX).
  • Custodiar QU (Qubic)? Sim — é o caso mais alinhado ao funil já decidido (QU → compra ECHONX). Na prática: usuário envia QU para depósito reconhecido pelo EchonX; o backend credita balance_qu na conta do perfil.
  • ECHONX + QU + stable na mesma tela? Sim na UX; no banco são saldos separados com regras de conversão entre eles.

O que definir no desenho inicial (evita refatorar depois)

  1. Tabela tipo wallet_balances(profile_id, asset_code, amount) — não uma coluna só echonx.
  2. asset_code: ECHONX, QU, USDC (etc.).
  3. Ledger de movimentos: compra, assinatura, depósito, saque, conversão QU→ECHONX.
  4. Regras por ativo: ECHONX não transferível (v1); QU/stable podem ter depósito/saque com KYC depois.

Cuidados

  • Stablecoin implica rede (Ethereum, etc.), compliance, e custódia real de tokens — mais trabalho que QU+ECHONX só no ecossistema Qubic.
  • QU custodial exige processo de depósito (endereço, confirmações, indexer) — pode começar manual/admin e automatizar depois.
  • Não confundir saldo interno (número no Postgres) com coins on-chain até a ponte de depósito/saque existir.
Recomendação de escopo v1: carteira multi-ativo no modelo de dados, mas produto mínimo = ECHONX + QU (stablecoin na v2 se precisar de preço em dólar estável sem volatilidade).

11. Pgledger + mudar de chain (Qubic → Solana): a carteira continua compatível?

Sim — se a carteira for ledger interno e a blockchain for uma camada plugável. Se o código misturar saldo no Postgres com lógica só Qubic, migrar para Solana vira reescrita. Com separação certa, a maior parte do pgledger permanece.

Duas camadas

Camada O quê Muda ao trocar Qubic → Solana?
Ledger (pgledger) Saldos por perfil, extrato, assinatura em ECHONX Não
Chain adapter Depósito/saque, confirmações, mint on-chain Sim (novo adaptador)

Resposta direta

Postgres primeiro maximiza compatibilidade. ECHONX pode ser só crédito interno; QU ou SOL são rampas de entrada via adaptador. Troca-se QubicAdapter por SolanaAdapter — não o ledger. Evite amarrar migrations do ledger a IDs ou QPI da Qubic.

12. Criar ECHONX na Solana: facilita a tokenomics?

Facilita a parte técnica de “cunhar um token” — não substitui o desenho econômico (emissão, preço, consumo, tesouraria). Para o EchonX, Solana é em geral mais rápida e barata que contrato no core Qubic; muda o funil se a entrada continuar sendo “QU compra ECHONX”.

O que “tokenomics” inclui (separar)

Bloco Conteúdo Solana ajuda?
Token on-chain SPL mint, supply, decimais, freeze/mint authority Muito — dias/semanas, ferramentas maduras (Metaplex, Anchor, etc.)
Economia do produto Preço em QU/USD, crédito não transferível, assinatura, queima, tesouraria Pouco — é decisão de negócio + pgledger (igual em qualquer chain)
Distribuição / vesting Alocações equipe, comunidade, liquidez Sim — padrões SPL + multisig (Squads) + lockups conhecidos
Integração EchonX Depósito SPL → crédito pgledger; pagar plano Médio — indexer RPC, custódia hot wallet, mesmo papel que Qubic adapter

Solana vs Qubic (para ECHONX on-chain)

Critério Solana (SPL) Qubic (core + QPI)
Tempo até token existir Curto (deploy programa/mint) Longo (PR, IPO, computors, epochs)
Custo / complexidade dev Ecossistema grande, docs, devs C++ restrito, poucos contratos de referência
Alinhamento “QU → ECHONX” Fraco — entrada natural seria SOL/USDC Forte — QU nativo, narrativa Qubic
Compatível com pgledger primeiro Sim (SolanaAdapter) Sim (QubicAdapter)
Crédito consumo off-chain (v1) Igual — Postgres não muda Igual

Avaliação para o seu caso (decisões já tomadas)

Vocês definiram: receita off-chain, ECHONX consumo, custodial, QU compra ECHONX, assinatura em QU e ECHONX.

  • Se ECHONX nasce só na Solana: tokenomics técnicas mais fáceis, mas o funil “QU → compra ECHONX” vira estranho — usuário precisaria bridge QU→Solana ou comprar ECHONX com SOL/USDC na Solana.
  • Modelo híbrido coerente: v1 = ECHONX só no pgledger (sem mint); v2 = SPL na Solana para quem quer “token de verdade”, saque/depósito; entrada principal USDC-SPL ou SOL; QU opcional via bridge ou fase Qubic depois.
  • Modelo Qubic-first: mantém narrativa; tokenomics on-chain mais lentas; pgledger igual.

Veredito

Solana facilita criar o ativo e mecânicas on-chain (mint, vesting, DEX, liquidez) — ótimo se o objetivo é lançar token cripto rapidamente. Não simplifica sozinha a tokenomics de produto EchonX (crédito não transferível, preço de planos, tesouraria custodial). Para o funil atual (QU → ECHONX), Solana só é “melhor” se vocês aceitarem mudar a rampa de entrada (ex. USDC/SOL na Solana) ou rodar dois ativos de entrada.

Recomendação estratégica: desenhe tokenomics de negócio no pgledger primeiro; escolha Solana para mint público se prioridade for velocidade/ecossistema; mantenha Qubic como opção de pagamento/parceiro se a marca Qubic for importante — não é obrigatório que o mint e a rampa sejam a mesma chain.

13. Direção acordada (estudos — economia primeiro)

Decisão de produto: construir a economia dentro da plataforma com pgledger antes de cripto on-chain. Primeiro produto comercial: assinaturas pagas com créditos ECHONX comprados via Stripe. Outros tipos de transação ficam para fases posteriores.

Princípios

  • Crédito ECHONX = unidade de consumo na plataforma (não transferível entre usuários na v1).
  • Carteira por perfil = saldo + extrato contábil (auditoria, suporte, disputas).
  • Stripe = rampa fiat para comprar pacotes de créditos; webhook confirma pagamento → credita pgledger.
  • Assinatura = debita créditos ECHONX (e/ou mantém registro de plano em subscriptions para direitos de produto).
  • On-chain (Qubic, Solana, QU) = fase futura; não bloqueia v1.

Fluxo v1 (assinatura)

  1. Usuário compra pacote de créditos ECHONX (Stripe Checkout).
  2. Webhook Stripe → EchonX credita wallet_balance + linha no ledger (purchase_stripe).
  3. Usuário escolhe plano Starter / Popular / Pro (preço em créditos ECHONX).
  4. Sistema debita créditos + ativa período de assinatura (subscription_debit).
  5. Limites de plano (perfis X, etc.) continuam checados como hoje — enforcement ainda a implementar onde faltar.

Esboço pgledger (Postgres)

Separar saldo atual de histórico imutável (contabilidade).

  • wallet_accounts — profile_id, asset_code (= ECHONX na v1), balance (cache derivado ou atualizado em transação).
  • wallet_ledger_entries — id, account_id, amount (+/-), reason, reference_type, reference_id, created_at, idempotency_key.
  • credit_packages — pacotes Stripe (ex. 1000 ECHONX por US$9).
  • subscription_entitlements ou estender subscriptions — plano ativo, valid_until, ligado ao perfil.

Motivos de ledger sugeridos: stripe_purchase, subscription_charge, admin_adjustment, refund (futuro). Toda mudança de saldo = 1 linha no ledger (append-only).

Fases sugeridas

Fase Escopo Fora de escopo
1 Migration pgledger + API creditar/debitar + UI saldo/extrato QU, Solana, P2P
2 Stripe: compra de pacotes → crédito ECHONX (webhook idempotente) Assinatura recorrente Stripe automática (opcional depois)
3 Checkout assinatura em créditos ECHONX + valid_until Outros produtos (tips, marketplace)
4+ Novos reason no ledger; QU/stable; on-chain espelho —

Alinhamento com o código hoje

  • Já existe subscriptions + planos em plans.ts + Stripe env no .env.example.
  • src/app/api/webhooks/stripe/route.ts tem TODO para persistir assinatura — na v1 pode primeiro creditar ECHONX e depois unificar com plano.
  • Audiopost / fila / likes não dependem do pgledger na fase 1.

Por que esta ordem é prudente

Valida receita e UX com ferramentas maduras (Stripe + Postgres) antes de IPO Qubic ou SPL Solana. Tokenomics de negócio (preço do crédito, quanto custa cada plano em ECHONX) fica explícita no ledger e pode mudar sem redeploy de contrato.

14. Créditos ECHONX vs dólar no Stripe — precisa ser 1:1?

Não. Enquanto não existir criptomoeda on-chain, crédito ECHONX é unidade interna (ponto de consumo). Vocês definem a tabela de conversão USD → créditos — como qualquer jogo ou SaaS com “coins”.

Três formas comuns (podem combinar)

Modelo Como funciona Prós
A — Pacotes fixos Stripe vende SKUs: “US$9 → 1.000 créditos”, “US$19 → 2.200 créditos” Simples na UI e no webhook; sem fórmula exposta
B — Taxa configurável créditos = USD × taxa (ex. 100 créditos por US$1) Fácil ajustar inflação/bônus; Stripe cobra valor em USD, backend calcula créditos
C — Preço do plano em créditos Assinatura não usa taxa na hora: Starter = 900 créditos/mês (fixo na config) Desacopla “preço do plano” da “promoção de pacotes”

Recomendação v1: A + C — pacotes Stripe na loja + tabela plan_monthly_credits para assinatura.

“Parear 1:1” — o que isso pode significar

  • 1 crédito = US$0,01 (100 créditos = US$1) — peg mental, não é obrigatório nem é “1 ECHONX = 1 USD” de cripto.
  • 1 crédito = 1 futuro token ECHONX — só definam isso no dia em que existir mint on-chain; até lá, crédito é só ledger.
  • 1 US$ do Stripe = 1 crédito — ruim (Popular US$19 = 19 créditos); números pequenos demais.

Exemplo alinhado aos planos atuais (US$9 / 19 / 29)

Taxa de referência: 100 créditos = US$1 (configurável em platform_config).

Plano Preço USD/mês (hoje) Custo em créditos/mês (C) Equivalente na taxa 100:1
Starter US$9 900 créditos 9 × 100
Popular US$19 1.900 créditos 19 × 100
Pro US$29 2.900 créditos 29 × 100

Pacotes Stripe (A) podem dar bônus: ex. pagar US$10 e receber 1.100 créditos (+10%) para incentivar compra maior que 1 mês.

Implementação no pgledger

  • credit_packages: stripe_price_id, usd_cents, credits_granted (explícito — não só fórmula).
  • plan_catalog: plan_tier, credits_per_period, period_days (= 30).
  • Webhook Stripe: credits_granted do pacote comprado → wallet_ledger_entries (+).
  • Ativar plano: debitar credits_per_period → ledger (−) se saldo suficiente.
  • Guardar idempotency_key = stripe_session_id para não creditar duas vezes.

Ajustes sem “quebrar” economia

  • Mudar taxa global só afeta novos pacotes; pacotes antigos já vendidos mantêm créditos prometidos.
  • Assinatura pode subir de 900 → 950 créditos/mês na config; quem já pagou período atual não re-debita até renovação.
  • Exibir sempre: “Créditos ECHONX são unidades de consumo na plataforma, não são criptomoeda negociável.”

Resposta direta

Não precisam parear 1:1 com um token futuro nem 1:1 com dólar literal. Definam uma regra de negócio (ex. 100 créditos por dólar) ou pacotes fixos na Stripe; preço de cada assinatura em créditos é uma segunda tabela. Isso é totalmente independente de existir moeda ECHONX on-chain depois.

15. Só créditos avulsos — perde o caráter de SaaS?

Não necessariamente — depende do que você vende. Crédito avulso é forma de pagar; SaaS é forma de empacotar o produto (plano, período, direitos). Podem coexistir.

Três modelos lado a lado

Modelo O que o usuário compra Parece SaaS? MRR previsível
SaaS clássico Assinatura US$19/mês (Stripe Subscription) Sim Alto
Só créditos avulsos Pacotes de créditos; gasta em “ações” (dias de plano, features) Fraco — parece prepago / wallet Baixo (compra esporádica)
Híbrido (recomendado) Plano Starter/Pro com direito mensal; paga em créditos ECHONX (comprados na Stripe) Sim — continua plano + renovação Médio/alto se renovar todo mês

O que você descreveu (créditos → assinatura)

Isso não é “só avulso”: o produto comercializável continua sendo assinatura (SaaS). O crédito é só meio de pagamento interno — como um “saldo na loja” antes de assinar. Característica SaaS preservada se existir:

  • plan_tier + valid_until (ou período mensal);
  • renovação explícita (todo mês debita X créditos ou avisa “recarregue”);
  • limites por plano (perfis X, Pro, etc.) — não “cada clique gasta crédito”.

Quando créditos avulsos sim enfraquecem SaaS

  • Usuário só compra pacotes quando acaba, sem plano nem data de renovação.
  • Cada feature tem micro-preço em créditos (pay-per-listen, pay-per-post) — vira marketplace, não assinatura.
  • Não há distinção Free / Starter / Pro — só saldo.
  • Métricas viram “venda de pacote”, não “assinantes ativos”.

Opinião / recomendação

Vender pacotes de créditos na Stripe não mata o SaaS, desde que o objeto vendido siga sendo “acesso ao plano por período”, não “créditos como produto final”.

  • Camada 1 (SaaS): entitlement — quem é Starter até quando.
  • Camada 2 (pagamento): créditos ECHONX debitados na renovação (ou Stripe USD em paralelo para quem preferir).
  • Camada 3 (opcional): top-up avulso — “recarregar carteira” entre meses, sem substituir o plano.

Mais tarde: Stripe Subscription em USD para quem não quer créditos + pacotes ECHONX para quem quer — mesmo plano, duas rampas.

Resumo em uma frase

Crédito avulso é a carteira; SaaS é a assinatura. Se a assinatura continuar existindo (mesmo paga em créditos), você mantém SaaS. Só perde se o negócio virar exclusivamente “compre créditos e gaste até acabar” sem planos nem renovação.

16. Débito automático mensal em créditos + Stripe só para comprar saldo

Sim, pode ser realizado. Fluxo acordado: compra créditos na Stripe → saldo na carteira → débito interno libera/renova o plano → todo mês o job tenta debitar de novo → se falhar, banner “sem saldo” e CTA para a página de pagamento.

Duas cobranças diferentes (não confundir)

Momento O que acontece Stripe?
Comprar créditos Checkout paga pacote → webhook credita pgledger Sim (cartão, Link, Apple/Google Pay, etc. conforme conta Stripe)
Ativar / renovar plano Debita X créditos ECHONX do saldo → estende current_period_end Não — é só Postgres (pgledger)

A renovação mensal não cobra o cartão de novo na v1 — só gasta o saldo que o usuário já comprou. O cartão entra de novo quando ele volta à página de pagamento e compra outro pacote.

Fluxo completo (v1)

  1. Compra: Stripe Checkout (modo payment, pacote de créditos) → checkout.session.completed → credita carteira (idempotente por session.id).
  2. Escolher plano: usuário seleciona Starter / Popular / Pro na UI de billing.
  3. Primeira ativação: transação debita plan_credits[plan]; se saldo ≥ custo → plan + current_period_end = now() + 1 month + ledger subscription_charge.
  4. Renovação automática (cron): diário (ou a cada hora) busca assinaturas com current_period_end <= now() e billing_mode = credits (ou equivalente).
  5. Tentativa de débito: mesma transação atômica: se saldo suficiente → debita + estende +1 mês; senão → status = past_due (ou mantém plano até fim do período já pago e depois rebaixa).
  6. Sem saldo: banner no app + e-mail opcional: “Saldo insuficiente — recarregue créditos” → link para /app/settings/billing (ou pricing) com pacotes Stripe.

Estados sugeridos em subscriptions

  • active — período pago, direitos do plano.
  • past_due — período expirou e débito falhou; limites do Free ou “somente leitura” até recarregar.
  • canceled — usuário cancelou renovação (não tenta debitar no próximo ciclo).

Campos úteis: billing_mode (credits | stripe_usd futuro), renewal_credits, last_debit_at, last_debit_error.

Meios de pagamento Stripe (v1)

Na compra de créditos, Stripe Checkout aceita o que estiver habilitado na conta: cartão, Link, Apple Pay, Google Pay, ACH (EUA), etc. Vocês controlam em Payment methods e no payment_method_types da sessão.

Por enquanto: só rampa fiat Stripe para entrada de créditos; renovação = débito interno. Assinatura recorrente Stripe em USD (cobrança automática no cartão todo mês sem passar pela carteira) fica como fase opcional paralela.

Ex.: 3 dias após current_period_end ainda com plano ativo mas banner “renove em breve”; depois past_due. Melhora UX sem cobrar cartão automaticamente.

Resposta direta

17. Quem decide a recorrência no cartão? Recarga automática de créditos

Vocês definem o produto; o cliente aceita explicitamente ao cadastrar o cartão num fluxo de assinatura / recarga automática. Desconto no cartão é regra sua (cupom Stripe); o cliente só aplica se vocês permitirem código promocional.

Dois “relógios” mensais (complementares)

Relógio Quem cobra O que faz
A — Recarga de créditos (fiat) Stripe no cartão salvo Todo mês (ou quando saldo baixo): credita ECHONX na carteira
B — Renovação do plano (SaaS) EchonX (pgledger) No current_period_end: debita créditos → +30 dias de plano

O cenário ideal que você descreveu usa os dois: A mantém saldo; B mantém entitlement. Se só existir B sem A, o usuário precisa comprar pacotes manualmente quando acabar o saldo.

Quem decide se é recorrente?

  • EchonX (produto): vocês oferecem SKUs distintos — ex. “Pacote 1000 créditos (único)” vs “Recarga mensal 1000 créditos (automático)”.
  • Cliente: escolhe e confirma no Checkout; cadastra cartão num fluxo com texto claro (“cobrança mensal até cancelar”).
  • Stripe: executa o que o SKU/modo pedir — payment (único) ou subscription (recorrente).
  • Desconto: vocês criam cupom/promo no Stripe; pode ser automático num plano ou o cliente digita código — isso é configuração sua, não decisão solta do gateway.

Compra avulsa com desconto não vira recorrente sozinha — só vira recorrente se o produto/checkout for subscription ou se o cliente ativar “recarga automática” (opt-in separado).

Como cadastrar cartão para compra mensal de créditos (recomendado)

Opção 1 — Stripe Subscription no pacote de créditos (mais simples)

  1. Produto Stripe: “Recarga mensal — 1900 créditos” (preço recorrente / mês).
  2. Checkout mode: subscription → cliente salva cartão; Stripe cobra todo mês.
  3. Webhook invoice.paid → credita pgledger (stripe_subscription_reload).
  4. Cron B debita plano se saldo ≥ custo do tier.

Opção 2 — Cartão salvo + cobrança off-session (mais flexível, mais trabalho)

  1. SetupIntent / Customer salva payment_method.
  2. Flag auto_reload_enabled + valor/pacote escolhido pelo cliente.
  3. Cron ou saldo < limiar → PaymentIntent com off_session: true.
  4. Falha de 3DS → e-mail + link para reautenticar (SCA Europa/BR conforme caso).

Para v1, a Opção 1 costuma bastar e alinha com “cliente cadastra cartão para recarga mensal”.

Alinhar valores (evitar surpresa)

  • Definir pacote recorrente ≥ custo mensal do plano em créditos (ex. Popular = 1900 créditos/mês → recarga Stripe de 1900 ou 2000).
  • Ou: na ativação do plano, sugerir “Ative recarga automática de X créditos” com um clique.
  • Se A creditar depois de B tentar debitar, usar ordem no cron: primeiro processar recargas Stripe pendentes / saldo, depois débito de plano (ou grace de alguns dias).

Cancelamento e portal

  • Cliente cancela recarga Stripe no Customer Portal → para de entrar crédito; plano pode cair em past_due quando B falhar.
  • Cliente cancela só renovação de plano → para de debitar B; saldo pode sobrar na carteira.
  • Deixar explícito na UI: “Recarga automática de créditos” ≠ “Plano Popular ativo” (dois toggles ou um pacote bundled).

Resumo

Quem decide o modelo recorrente: vocês (catálogo). Quem autoriza cobrar de novo: o cliente (aceite no Checkout / portal). Fluxo desejado: cartão salvo → Stripe recarrega créditos todo mês → ledger debita plano → +30 dias. Compra única com desconto não substitui isso; é outro SKU.

20. Sync global só para “online” — reduz custo?

Sim, o custo reduz em relação ao sync global “sempre todos os perfis”, porque passa a depender de Uonline (perfis X únicos com pelo menos um ouvinte online) em vez de U (todos). É possível no EchonX: já existe profiles.last_seen_at atualizado pelo heartbeat (~60s com Audiopost ativo).

Três modelos (relembrar)

Modelo Depende de usuários online? Variável de custo (timeline / 30 min)

O que significa “online” no EchonX (hoje)

  • POST /api/presence/heartbeat atualiza profiles.last_seen_at do usuário logado.
  • Disparado pelo cliente com Audiopost (~a cada 60s) enquanto o painel de listening está ativo.
  • Admin considera online quem tem last_seen_at nos últimos 2 minutos.

Para o cron de sync, usar janela parecida (ex. 5–30 min) para não cortar usuário entre dois heartbeats.

Definição de Uonline

Uonline = contagem de perfis external_x distintos em want_to_hear para os quais existe pelo menos um listener_x_user_id cujo perfil nativo tem last_seen_at > now() - intervalo_online. 

-- U_online (ex.: online = últimos 30 min)
select count(distinct w.target_profile_id) as u_online
from public.want_to_hear w
join public.profiles target on target.id = w.target_profile_id and target.kind = 'external_x'
join public.profiles listener on listener.id = w.listener_x_user_id
where listener.last_seen_at > now() - interval '30 minutes';

A cada 30 min o job global chama getLatestXPosts só para esses Uonline perfis (deduplicado, como no admin), e enfileira para todos os seguidores — inclusive os que estão offline desse mesmo @ (decisão de produto abaixo).

Decisão §4 (produto): melhor qualidade exige plano Library pago. Free nunca conta bytes; não há bloqueio de tempo no Web Speech.

30.4 Quota — o que conta e onde grava

  • Unidade: bytes UTF-8 do texto enviado à API Fish (após prepareMirroredPostForSpeech).
  • Campos DB: period_byte_quota (teto do ciclo), bytes_consumed (uso acumulado), current_period_end.
  • Débito: após síntese Fish bem-sucedida em POST /api/library/tts → RPC increment_library_bytes_consumed(owner, delta).
  • Pré-check: antes de chamar Fish, servidor recusa se bytes_consumed + tamanho_segmento > period_byte_quota → HTTP 403 LIBRARY_QUOTA_EXHAUSTED.
  • Renovação: quando Stripe confirma novo ciclo (current_period_end muda, mesmo tier) → bytes_consumed = 0, quota = teto mensal do plano.
  • Não acumula: bytes não usados no mês não passam para o ciclo seguinte.

30.5 Upgrade no meio do ciclo — regra A (só delta)

Confirmado: ao subir de tier no mesmo período de faturação, o utilizador ganha apenas a diferença entre as quotas mensais dos dois planos — não a quota inteira do plano novo em cima da antiga.
TransiçãoDelta de quota (exemplo)bytes_consumed
Starter → Popular +300.000 (600k − 300k) Mantém o já consumido
Popular → Pro +600.000 (1,2M − 600k) Mantém
Starter → Pro +900.000 (1,2M − 300k) Mantém

Exemplo: Starter 300k, já consumiu 280k → sobram 20k Fish. Upgrade para Popular → teto efetivo 300k + 300k delta = 600k no ciclo; consumido continua 280k → ~320k Fish restantes.

Código: libraryQuotaDeltaOnUpgrade + applyLibraryPlanTransition em library-quota-policy.ts · sync webhook em library-stripe-sync.ts.

30.6 Stripe — checkout por plano (sem proration na mesma sessão)

Confirmado: cada tier Library tem o seu Price ID no Stripe. Mudar de plano = novo Checkout Session (POST /api/billing/library/checkout), não alteração automática de preço na subscrição antiga com proration.
  • STRIPE_PRICE_LIBRARY_STARTER, STRIPE_PRICE_LIBRARY_POPULAR, STRIPE_PRICE_LIBRARY_PRO
  • Metadata na subscrição: billing_product=library, plan_tier=library-starter|library-popular|library-pro, owner_x_user_id
  • Retorno: ?checkout=library-success / ?checkout=library-cancelled
  • Se já existir outra subscrição Stripe Library ativa, o webhook cancela a subscrição anterior ao ativar a nova (evita duplicar cobrança).

30.7 Downgrade — corta na hora

Confirmado: ao passar para um tier menor, o teto period_byte_quota passa imediatamente ao valor mensal do plano novo — não espera o fim do ciclo.
Situação após downgradeComportamento Fish
bytes_consumed ≤ novo teto Fish continua até esgotar o novo teto
bytes_consumed > novo teto Fish bloqueado de imediato; Web Speech ilimitado; upgrade ou renovação para voltar a Fish

Exemplo: Popular 600k com 500k consumidos → downgrade Starter (300k) → quota efetiva 300k, consumido 500k → Fish bloqueado.

30.8 Library Pro esgotado — só renovação

Confirmado: no tier Pro (último), não há upgrade nem pacotes de horas extra. Com quota a zero antes do fim do ciclo, o utilizador deve aguardar o pagamento recorrente Stripe; após confirmação (customer.subscription.updated com novo período), o webhook repõe quota e zera consumo.
  • UI: mensagem “renova em {data}” na barra (LibraryQuotaStrip) e no preview do player.
  • Link de billing: secção #library-premium · query ?library=library-popular para destacar card de upgrade (Starter/Popular apenas).
